## Symptoms
- **Expected behavior**: Spring Boot tests (like `AiTest`) should load the ApplicationContext successfully and execute.
- **Actual behavior**: Context fails to load with `java.lang.IllegalStateException: Failed to introspect Class [com.casso.milktea.controller.PaymentWebhookController]`.
- **Error messages**: `java.lang.NoClassDefFoundError: TeaShopBot` and `ClassNotFoundException: TeaShopBot`.
- **Reproduction**: Run `mvn test -Dtest=AiTest`.

## Evidence Gathered
- Investigated `PaymentWebhookController.java` and `TeaShopBot.java`. The `TeaShopBot` class exists, has the correct package `com.casso.milktea.bot.TeaShopBot`, and is properly imported.
- Ran `javap -c -p` on the generated `PaymentWebhookController.class` file in `target/classes`. The bytecode showed:
  `String Unresolved compilation problems: \n\tThe import com.casso.milktea.bot.TeaShopBot cannot be resolved\n\tTeaShopBot cannot be resolved to a type`
- This indicates the `.class` file was generated by an IDE's error-tolerant incremental compiler (e.g. Eclipse / VS Code Java Extension) during a moment when `TeaShopBot` had a syntax error or its path was broken.
- Because `PaymentWebhookController.class` had a timestamp newer than the source file, a standard `mvn compile` or `mvn test` skipped recompiling it ("Nothing to compile - all classes are up to date").

## Root Cause
The `target/classes` directory was corrupted with a broken `.class` file generated by an IDE compiler that included a hardcoded `java.lang.Error` stub for an unresolved type. Maven saw the `.class` file was newer than `.java` and skipped recompilation, causing the JVM to throw `NoClassDefFoundError` at runtime when it tried to introspect the broken class.

## Fix
Ran `mvn clean compile` to delete the corrupted `target` directory and force a clean rebuild from the source files. After cleaning, `mvn test -Dtest=AiTest` passed successfully without any Context Load errors.
